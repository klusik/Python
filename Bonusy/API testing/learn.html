1. The Big Picture

Imagine you are looking at a catalog of products in a store. You want to:

  • see all products,
  • search for a word like “coffee”,
  • show only one category, like Electronics,
  • sort by price,
  • open a single item to inspect its details.

This project shows how a browser page can ask a Python program for that information. The browser is the client. The Python server is the server.

Simple version: the browser asks questions, the server answers with data.

2. What HTML, CSS, and JavaScript Are

These three technologies work together in web pages:

HTML

HTML is the structure. It says what elements exist on the page: headings, paragraphs, buttons, inputs, and sections.

CSS

CSS is the appearance. It controls colors, spacing, layout, borders, fonts, and how the page looks on different screens.

JavaScript

JavaScript is the behavior. It can read user input, send network requests, process returned data, and update the page without reloading the whole site.

In this project, all three live together in index.html. That makes the file long, but it is easier to study because everything is in one place.

3. What an API Is

API stands for Application Programming Interface. That sounds complicated, but the idea is simple.

An API is a way for one program to ask another program for data or services using a defined set of rules.

Think of an API like a waiter in a restaurant. You do not go into the kitchen yourself. You tell the waiter what you want, and the kitchen sends back the result.

In this project:

  • The browser page is the person at the table.
  • The Python server is the kitchen.
  • The API is the waiter who carries the request and the response.

4. What a GET Request Is

A GET request is a request for data. It is the most common type of request when you want to read information from a server.

For example, when the browser visits this URL: 

http://localhost:8484/api/products

it is saying, “Please give me the list of products.”

GET is useful because the request is visible in the URL. That means you can copy it, paste it, bookmark it, and test it directly in the browser.

GET is best for reading data. It is not used for changing data in this project.

5. What Query Strings Are

A query string is the part of a URL after the ?. It lets you send extra details with a GET request. 

http://localhost:8484/api/products?category=Electronics&sort_by=price

In this example:

  • category=Electronics tells the server to only show Electronics.
  • sort_by=price tells the server to sort the result by price.

Each pair is called a parameter. The part on the left is the name, and the part on the right is the value.

Parameter name

category, search, page, sort_by

Parameter value

Electronics, coffee, 2, price

This is one of the most important ideas in web development: the browser can send instructions to the server using a URL.

6. What JSON Is

JSON stands for JavaScript Object Notation. It is a text format for storing and exchanging data.

JSON looks similar to JavaScript objects, but it is really just text that both servers and browsers can read. 

{
  "id": 3,
  "name": "BaristaPro Coffee Grinder",
  "price": 64.0,
  "rating": 4.2
}

The server sends JSON because it is lightweight, easy to read, and very common in APIs.

The browser uses fetch() to get JSON, then converts it into JavaScript objects.

7. What the Python Server Does

The server lives in server.py. It does not show a webpage. Its job is to listen for requests and return JSON responses.

When a request arrives, the server:

  1. looks at the path in the URL,
  2. reads any query parameters,
  3. decides which endpoint should handle the request,
  4. filters or processes the data,
  5. sends back a JSON response.

Why the data is inside Python

The sample product list is stored directly in a Python variable. That keeps the project simple and avoids the distraction of a database.

In a larger app, the data might come from SQLite, PostgreSQL, MongoDB, or an external service.

8. What the Browser Page Does

The browser page is written in index.html. It contains the controls, the display area, and the JavaScript that talks to the server.

The JavaScript does three important jobs:

  • reads values from inputs and dropdowns,
  • builds a request URL,
  • updates the page with the returned data.

A good way to think about it: the browser page is the front desk, and the API is the back office.

The page is not “hardcoded” to one result. It asks the server again each time you change a filter.

9. The Full Workflow

Here is the full loop from user action to on-screen result:

  1. You type a search term like coffee.
  2. JavaScript reads that text from the input box.
  3. JavaScript builds a URL such as /api/products?search=coffee.
  4. The browser sends a GET request to the Python server.
  5. The server filters the sample products.
  6. The server sends back JSON.
  7. JavaScript receives the JSON and renders cards, stats, and the raw response.

That loop is the heart of many real web applications.

If you understand this loop, you understand the basic client-server model used by most APIs.

10. The Endpoints in This Project

An endpoint is just a specific URL path that does one job.

/api/products

Returns the list of products, with filtering, sorting, and pagination.

/api/products/3

Returns one product by ID.

/api/categories

Returns the available categories for dropdowns.

/api/brands

Returns the available brands for dropdowns.

/api/stats

Returns summary information about the filtered result set.

/api

Shows API information and example URLs.

The important idea is that different URLs can represent different pieces of data or different actions.

11. How to Experiment Safely

The best way to learn this project is to change one thing at a time.

  • Open the browser devtools and watch the Network tab.
  • Change one filter and note the new request URL.
  • Compare the raw JSON preview with the product cards.
  • Open the API URL directly in a browser to see the JSON without the UI.
  • Try an invalid ID, like /api/products/999, and read the error response.
Learning becomes much faster when you actively test the system instead of only reading it.

12. What to Learn Next

Once you are comfortable with this project, good next topics are:

  • HTTP methods beyond GET, like POST and DELETE,
  • working with a real database,
  • validation of user input,
  • error handling patterns in APIs,
  • JavaScript event handling,
  • fetching data asynchronously with promises and async/await.

This project gives you the foundation for those topics.

If you want, the next step can be a version with POST requests so you can create new products from the browser.
